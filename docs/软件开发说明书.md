# 软件开发说明书

## 1. 项目概述

### 1.1 项目背景
高校课堂教学数字化过程中，传统教学管理系统多聚焦课程与成绩，缺少课堂过程监控与学习状态反馈。本项目构建一个面向课堂过程的数据化平台，实现教学管理与学习行为检测一体化。

### 1.2 项目建设目标
- 建设管理员端、教师端、学生端三端完整系统
- 实现课程、会话、签到、测验、统计等基础教学闭环
- 实现课堂行为实时采集、分析、落库、展示闭环

### 1.3 项目定位
定位为“可运行、可联调、可测试、可展示”的课程设计/毕业设计级在线课堂管理系统。

### 1.4 项目核心特色
- 三端职责清晰、权限边界严格
- 行为检测模块独立分层，可扩展到AI
- 统计分析基于真实数据库记录，不依赖假数据

## 2. 可行性分析

### 2.1 技术可行性
- 前端采用 Vue3 + Vite，生态稳定
- 后端采用 Spring Boot + MyBatis Plus，开发效率高
- MySQL 满足关系型业务需求
- JWT 认证方案成熟

### 2.2 经济可行性
- 采用开源技术栈，软件授权成本低
- 硬件需求常规，部署成本可控

### 2.3 运行可行性
- 系统支持本地或局域网部署
- 前后端分离便于扩展与运维

### 2.4 开发可行性
- 模块边界清晰，支持分阶段推进
- 功能可逐步增强，具备持续迭代能力

## 3. 需求分析

### 3.1 业务需求分析
- 管理员需要全局教学与行为数据管理能力
- 教师需要课程与课堂活动组织能力
- 学生需要选课、上课、签到、测验与个人学习反馈能力

### 3.2 角色需求分析
- 管理员：全局可视、审核、配置、统计
- 教师：仅操作本人课程及课堂数据
- 学生：仅访问本人学习数据

### 3.3 功能需求分析
- 用户管理
- 课程管理与审核
- 选课与课堂会话
- 签到与测验
- 行为检测
- 统计分析
- 认证与权限控制

### 3.4 非功能需求分析
- 响应时间可接受
- 权限控制严格
- 接口返回规范统一
- 支持答辩演示稳定运行

### 3.5 核心检测需求分析
系统需持续采集课堂行为并进行规则分析，自动输出状态标签与描述，并支持教师、管理员、学生按权限查看。

## 4. 系统总体设计

### 4.1 总体架构设计
采用前后端分离架构：
- 前端负责页面交互、路由控制、图表展示
- 后端负责业务逻辑、权限控制、数据处理
- MySQL 负责持久化存储

### 4.2 技术架构设计
- 前端：Vue3 + Router + Pinia + Axios + ECharts
- 后端：Spring Boot + MyBatis Plus + JWT + Knife4j
- 数据：MySQL

### 4.3 功能模块划分
- 认证与权限模块
- 用户模块
- 课程与审核模块
- 选课与会话模块
- 签到模块
- 测验模块
- 行为检测模块
- 统计分析模块

### 4.4 核心业务流程
1. 教师创建课程 -> 管理员审核
2. 学生选课 -> 加入课程
3. 教师创建会话 -> 发起签到与测验
4. 学生进入在线课堂 -> 行为上报
5. 后端规则分析 -> 检测记录落库
6. 三端按权限查看记录与统计

### 4.5 三端职责划分
- 管理员：全局管理与决策
- 教师：教学活动组织与班级分析
- 学生：学习参与与个人反馈

以用户、课程、会话为主轴，签到/测验/行为检测围绕会话关联，形成课堂过程数据链。

## 6. 详细设计

### 6.1 登录与权限模块设计
- `AuthController + AuthService` 完成登录与当前用户信息
- `AuthInterceptor` 完成 Token 解析与角色校验
- `@RequireRole` 控制接口访问角色

### 6.2 用户管理模块设计
- 管理员可执行增删改查、重置密码、状态管理
- 密码采用哈希加盐存储

### 6.3 课程与审核模块设计
- 教师创建/编辑课程后进入待审核状态
- 管理员审核更新课程状态并记录审核日志

### 6.4 选课与课堂会话模块设计
- 学生仅可选修审核通过课程
- 课堂会话作为签到/测验/行为检测的统一关联实体

### 6.5 签到模块设计
- 教师发起签到任务
- 学生签到生成签到记录，支持迟到判定

### 6.6 测验模块设计
- 教师创建测验与题目
- 学生提交答案后自动评分并记录

### 6.7 行为检测模块设计
- 前端定时心跳上报
- 后端规则分析生成状态与活跃度
- 原始数据与结果分表存储

### 6.8 统计分析模块设计
- 管理员：全局概览、异常分布、趋势
- 教师：签到率、参与率、行为分布、异常学生
- 学生：个人签到/测验/行为趋势

## 7. 行为检测核心设计说明

### 7.1 行为采集机制
通过学生在线课堂页面监听点击、输入、焦点状态，结合定时器形成采集数据包。

### 7.2 心跳上报机制
默认每15秒调用 `/api/student/behavior/report`，保证课堂过程持续产出检测记录。

### 7.3 活跃度计算方式
以基础活跃分为底，结合交互行为加分、失焦与离线扣分，输出 0-100 分。

### 7.4 状态判定规则
通过签到、焦点、交互、测验参与等规则映射到状态标签与风险级别。

### 7.5 异常识别逻辑
当多项异常规则同时触发时，标记 `HIGH_RISK` 与 `学习状态异常`。

### 7.6 检测结果记录逻辑
每次上报写两张表：
- `behavior_raw_data`
- `behavior_record`

### 7.7 未来AI扩展方案
- 在 `BehaviorAnalyzeService` 注入模型策略接口
- 支持视频特征输入
- 支持第三方AI服务调用
- 支持风险通知事件化发布

- 网络层：Axios 封装

### 9.2 页面划分

- 管理员页：用户、课程、审核、会话、行为、统计、权限

## 11. 部署与运行说明

### 11.1 环境要求
- JDK 21+
- Maven 3.9+
- Node 18+
- MySQL 8.0+

### 11.2 部署步骤
1. 初始化数据库
2. 启动后端
3. 启动前端

### 11.3 启动步骤
- 后端：`mvn spring-boot:run`
- 前端：`npm install && npm run dev`

### 11.4 常见问题说明
- 若 maven/npm 默认缓存目录无权限，需切换缓存目录或使用管理员权限
- 若登录报 JWT 密钥长度错误，检查 `application.yml` 中 `app.jwt.secret` 是否 >= 32 字节
